# Gas Can 0.1.6 Release Design

## Purpose

Ship the CLI version flags as Gas Can `0.1.6` by adding the repository's
required patch-version bump to PR #17, squash-merging that PR, and pushing a
verified annotated signed `v0.1.6` tag on the exact squash commit. After the
tag is verified on `origin`, use the repository's release driver to build,
sign, notarize, publish the GitHub release, and update the Homebrew tap.

## Version Bump

The bump happens on `feat/cli-version`, never directly on `main`. Change exactly
the version-bearing release files prescribed by the runbook:

- `crates/gascan-apple/Cargo.toml`
- `crates/gascan-core/Cargo.toml`
- `crates/gascan-e2e/Cargo.toml`
- `crates/gascan-proto/Cargo.toml`
- `crates/gascan/Cargo.toml`
- `crates/gascand/Cargo.toml`
- `Cargo.lock`, regenerated with `cargo update --workspace --offline`
- `README.md`
- `docs/release/macos-checklist.md`

The six manifests change only their package version from `0.1.5` to `0.1.6`.
The README and release checklist change their concrete release examples from
`0.1.5` to `0.1.6`. The lockfile update changes only the six workspace-member
package entries.

Do not change `scripts/Cargo.lock`: its `0.1.5` entry belongs to an unrelated
third-party dependency. Do not change the versions in
`tests/release/release-script-contract.sh`: they are arbitrary inputs to stubbed
contract cases.

Commit the bump before running release contracts because contract tests clone
`HEAD` for part of their verification.

## Bump Verification

Before updating PR #17, verify all of the following on the committed branch:

- Cargo metadata reports `0.1.6` for the `gascan` package.
- `cargo check --locked --workspace --all-targets` succeeds.
- Every `tests/release/*-contract.sh` script succeeds.
- The exact `--version`/`-V` process test reports `gascan 0.1.6` without daemon
  state.
- The full workspace test suite succeeds.
- Formatting, Clippy with warnings denied, and `git diff --check` are clean.
- The branch contains no unintended version replacements.

Push the bump commit to `feat/cli-version`. PR #17 must report a mergeable state
and no failing required checks before merge. A pending required check is waited
for rather than bypassed.

## Squash Merge

Squash-merge PR #17 into `main`. Do not create a merge commit or rebase-merge.
After GitHub reports the PR merged:

1. record the squash commit returned by GitHub;
2. fast-forward the local `main` checkout from `origin/main`;
3. verify local `main` HEAD equals the recorded squash commit;
4. verify Cargo metadata at that commit reports `0.1.6`; and
5. verify the committed tree contains the CLI version feature and intended
   nine-file bump.

Failure at any post-merge verification stops the workflow before tag creation.

## Signed Tag

Before creating a tag, confirm `v0.1.6` is absent from both local refs and the
`origin` remote. An existing local or remote tag is never moved, replaced, or
deleted automatically.

Create the tag at the verified `main` HEAD with:

```text
git tag -s v0.1.6 -m 'Gas Can 0.1.6'
```

No unsigned or lightweight fallback is permitted. If signing fails, stop and
report the signing blocker with no tag push. Before push, verify:

- `git cat-file -t refs/tags/v0.1.6` reports `tag`;
- `git verify-tag refs/tags/v0.1.6` succeeds; and
- `refs/tags/v0.1.6^{}` resolves exactly to local `main` HEAD.

Push only `refs/tags/v0.1.6`. Then query the remote tag ref and prove its tag
object ID equals the local tag object ID. Do not start release automation until
that comparison succeeds.

## Automated Release

Use the existing release driver rather than reproducing its individual package,
publication, and tap-update steps:

```text
./packaging/macos/release.sh 0.1.6 --check
./packaging/macos/release.sh 0.1.6
```

Run `--check` first. It must report that all release preconditions pass before
the live command is authorized to run. The preflight verifies the toolchain,
workspace version, clean release inputs, signed tag, GitHub state, signing
identities, notarization profile, and Homebrew tap, and renders the cask as a
non-mutating probe.

The live command owns the remaining release flow: it checks out the tag,
builds, signs, notarizes, verifies and publishes the package and GitHub release,
renders and validates the cask, commits and pushes the tap update, and restores
the original checkout. Success requires the driver to exit successfully and
the published GitHub release and tap update to be observable at `0.1.6`.

Do not replace a failed driver step with ad hoc manual publication. Preserve
the driver's recovery output and stop for operator review, especially if it
reports that the GitHub release is already live and only the tap update remains.

## Cleanup

After the automated release and its remote verification:

- remove the owned `.worktrees/cli-version` worktree from the main checkout;
- prune worktree metadata;
- delete the local `feat/cli-version` branch; and
- delete the remote feature branch if the squash merge did not remove it.

The tracked `.superpowers/sdd/progress.md` modification in the worktree is
controller bookkeeping. It is never committed and may be discarded only when
the owned worktree is removed after the release tag is verified.

## Failure Boundaries

- A failed bump test or PR check stops before merge.
- A merge verification failure stops before tag creation.
- An existing tag or signing failure stops before tag push.
- A local tag that fails verification is not pushed and is left for explicit
  operator inspection; it is not silently recreated.
- A remote tag mismatch is reported and never force-pushed.
- A failed `release.sh --check` stops before any build or publication.
- A failed live release follows the driver's stage-specific recovery guidance;
  no release, tag, or tap state is deleted or force-pushed automatically.
